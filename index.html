Git 基础

Git 是分布式版本控制系统。每个开发者本地都有完整历史，因此可以离线提交、创建分支、回滚版本，再与团队远程仓库交换修改。

分布式 vs 集中式

集中式依赖中央服务器；分布式允许每个本地仓库保存完整历史，容错性和协作灵活性更高。

核心术语

仓库、工作区、暂存区、提交、分支、远程仓库，是理解 Git 操作链路的基础。

多人协作流程

团队通常采用集中式提交、功能分支或 Fork + Pull Request。课程项目可从功能分支开始，正式团队建议配合 Pull Request 和代码评审。

集中式

所有人直接向共享分支提交，简单但容易冲突。

功能分支

每个功能独立分支开发，稳定后合并到主分支。

Fork + PR

适合开源或跨团队协作，权限边界更清晰。

main student-a / feature review + merge
常用协作命令
git clone <repo>
git pull
git add .
git commit -m "message"
git push
冲突处理建议

先 `git pull` 获取最新修改，打开冲突文件，保留正确内容并删除冲突标记，再 `git add` 和 `git commit`。团队中应频繁拉取、提交信息清晰、分支命名如 `feature/login`、`fix/upload-error`。

同步到本地 NAS

NAS 可作为局域网内的 Git 裸仓库，用于备份、离线协作和提升内网访问速度。推荐在 NAS 上使用 bare repository，开发者本机只作为工作仓库。

NAS 裸仓库同步
git init --bare /path/to/NAS/repo.git
git remote add nas <NAS路径>
git push nas master
git pull nas master
权限

确保团队成员对 NAS 仓库目录有读写权限，避免一个用户创建后其他用户无法推送。

SSH

如果通过 SSH 访问 NAS，可配置公钥登录，并在远程地址中使用 `user@nas:/path/repo.git`。

同步到 GitHub

GitHub 适合远程备份、代码评审、Issue 管理和跨网络协作。HTTPS 上手简单，SSH 更适合长期开发环境。

冲突如何解决？

打开冲突文件，比较 `HEAD` 与 incoming change，保留正确内容，删除冲突标记后重新提交。

提交后发现错误如何回滚？

未推送可用 `git reset` 调整历史；已推送到共享分支时更建议用 `git revert` 生成反向提交。

NAS 与 GitHub 不一致怎么办？

先确认哪个远程最新，使用 `git fetch --all` 查看差异，再从可信远程拉取并推送到另一个远程。

大文件如何管理？

视频、模型、数据集等大文件建议使用 Git LFS，避免普通 Git 仓库体积快速膨胀。
